## Every release

- Refresh subprojects/pcg if any changes were made to the deckhandler pcg subproject
  `meson wrap promote subprojects/deckhandler/subprojects/pcg`
  https://github.com/mesonbuild/meson/issues/12489

- Pin the git subprojects so the bundled source is reproducible: each
  subprojects/<name>.wrap must use [wrap-git] with revision = vX.Y.Z (a
  tag/commit, NOT a branch or HEAD). The pinned .wrap is the contract.
  In the dev tree, subprojects/canfigger is a manual clone (gitignored) that
  can sit ahead of the pin or carry local edits, and
  `meson dist --include-subprojects` would bundle that WORKING TREE. Building
  the dist from a fresh clone (see below) avoids this entirely: a clean clone
  has no manual subproject checkout, so meson re-fetches each subproject at its
  pinned .wrap revision.

- Ensure all CI workflows are passing on trunk
- Checkout release branch
- Update ChangeLog: set the release date and version (replace "in progress")
- Bump version in meson.build (remove -dev suffix)
- Update po and pot files:
    meson compile dealers-choice-pot
    ninja dealers-choice-update-po
- Commit the above changes (ChangeLog + version)
- Build the dist from a fresh, clean clone -- NOT the dev tree -- so the
  tarball matches the committed state and the pinned subprojects:
    git clone <repo> ~/src/DealersChoice/dist
  Use the local path (git clone /home/.../dealers_choice ...) to include
  commits not yet pushed; a GitHub URL works only once trunk is pushed. The
  clone is pristine (tracked files only) with no manual canfigger checkout, so
  the bundled subprojects come from the pinned .wraps.
    cd ~/src/DealersChoice/dist/dealers_choice
- meson setup _build-release
- meson dist --include-subprojects (runs tests)
- Strip dev-only dirs from the dist tarball and refresh its sha256sum:
    cd _build-release && ../scripts/dist-strip-packaging.sh
  (removes the entries in scripts/dist-strip-list.txt -- .gitignore,
   .clang-format, .github, coverity, packaging -- and rewrites
   meson-dist/<pkg>.tar.xz.sha256sum)

- push to start testing

- mv ~/src/DealersChoice/dist/dealers_choice/_build-release/meson-dist/dealers-choice-0.0.*.tar.xz* \
      ~/src/DealersChoice/

- Update packaging; make sure they build (PKGBUILD, Slackbuild)
- Amend the commit with any changes, push
- Update the screenshots before posting the release
- Draft new GitHub release, paste the ChangeLog entry as notes

    *Attach source tarball and sha256sum*

- Trigger workflow dispatch on Windows workflow, use tag for target and enter
  version (without 'v')

- Same trigger for Debian package, from the Linux workflow

- Close milestone (https://github.com/Dealer-s-Choice/dealers_choice/milestones)

## Slackbuild

Use slackware-full:latest. Stock Slackware now ships meson >= 1.11, so the
python3-meson-opt SBo package is no longer needed.

## Check
sbo-maintainer-tools (sbolint and sbopkglint for SBo maintainers)

These are "lint" tools to make life easier for SlackBuilds.org
maintainers and admins:

sbolint: checks your SlackBuild, README, .info file, and slack-desc.
sbopkglint: checks your package, after it's built.
sbofixinfo: attempts to fix common problems with SBo .info files.
pre-commit-sbolint: git hook you should use, that automatically calls
sbolint and stops you from committing bad code.
sbodl: downloads source URLs, optionally using the SBo source archive.


# Post-release

- Bump version in meson.build to next -dev (e.g. 0.x.(y+1)-dev)
- Add new "in progress" section to top of ChangeLog
- Commit: "Begin development of 0.x.(y+1)"
