## Every release

- Ensure all CI workflows are passing on trunk
- Checkout release branch
- Update ChangeLog: set the release date and version (replace "in progress")
- Bump version in meson.build (remove -dev suffix)
- Update po and pot files:
    meson compile dealers-choice-pot
    ninja dealers-choice-update-po
- Comment out in meson.build:
    'b_sanitize=address,undefined',
    'b_lundef=false'
- Commit the above changes
- meson setup _release
- meson dist --include-subprojects (runs tests)

- push to start testing

- mv ~/src/DealersChoice/dealers_choice/_release/meson-dist/dealers-choice-0.0.*.tar.xz* \
      ~/src/DealersChoice/

- Update packaging; make sure they build (PKGBUILD, Slackbuild)
- Amend the commit with any changes, push
- Draft new GitHub release, paste the ChangeLog entry as notes
    Attach source tarball and sha256sum

- Trigger workflow dispatch on Windows workflow, use tag for target and enter
  version (without 'v')

- Close milestone (https://github.com/Dealer-s-Choice/dealers_choice/milestones)

## Check
sbo-maintainer-tools (sbolint and sbopkglint for SBo maintainers)

These are "lint" tools to make life easier for SlackBuilds.org
maintainers and admins:

sbolint: checks your SlackBuild, README, .info file, and slack-desc.
sbopkglint: checks your package, after it's built.
sbofixinfo: attempts to fix common problems with SBo .info files.
pre-commit-sbolint: git hook you should use, that automatically calls
sbolint and stops you from committing bad code.
sbodl: downloads source URLs, optionally using the SBo source archive.


# Post-release

- Uncomment in meson.build:
    'b_sanitize=address,undefined',
    'b_lundef=false'
- Bump version in meson.build to next -dev (e.g. 0.x.(y+1)-dev)
- Add new "in progress" section to top of ChangeLog
- Commit: "Begin development of 0.x.(y+1)"
